# Biometric Profile Implementation Plan

## Recommended Approach

The safest design for this SaaS is a three-layer flow: collect typing samples in the browser, transform them into compact behavioral features, and persist only aggregated biometric profiles on the backend. Do not store raw keystrokes as source-of-truth data in MongoDB. The frontend should own sample collection and progress feedback, the backend should own validation and persistence, and MongoDB should store only the stable profile state plus short summaries for dashboard use. That gives you a clean separation between onboarding, live scoring, and later drift detection.

The best fix is to make the profile-build path explicit and deterministic: the user types in the profile-build page, the browser collector accumulates samples, the frontend sends a structured payload when a sample threshold is reached, the backend validates the active session, and MongoDB upserts a per-user `biometric_profiles` document keyed by `user_id`, `tenant_id`, and `site_id`. A small summary may also be mirrored into `users.biometric_profile` for quick UI reads, but the real behavioral state should live in `biometric_profiles`.

The goal is to make sure the `/biometric/profile` route reliably writes the stable profile state, that the request/response schema matches the frontend payload, and that MongoDB stores a per-user biometric profile document with the expected fields. Please update [backend/main.py](../backend/main.py), [backend/database.py](../backend/database.py), and [backend/models.py](../backend/models.py), and also review [backend/tests/test_integration.py](../backend/tests/test_integration.py) if test coverage needs to be aligned with the new profile shape. The implementation should keep session validation strict, use `user_id` from the active session rather than trusting the client, upsert the profile by user identity, and make sure the stored document contains `sample_count`, `last_drift`, `adaptive_threshold`, `selected_features`, `feature_means`, `avg_theta`, `avg_h_exp`, `avg_dwell`, `avg_flight`, `avg_speed`, `avg_jitter`, `avg_accel`, `avg_rhythm`, `samples`, `created_at`, and `updated_at`.

Use this prompt if you want Claude to repair the UI and client-side collection path only. The goal is to make the profile-build page generate and submit stable sample batches, show accurate progress, and stop pretending the profile exists when no backend write happened yet. Please update [src/pages/ProfileBuildPage.jsx](../src/pages/ProfileBuildPage.jsx), [src/context/AuthContext.jsx](../src/context/AuthContext.jsx), [src/services/biometricCollector.js](../src/services/biometricCollector.js), [src/services/biometrics.js](../src/services/biometrics.js), and [src/services/api.js](../src/services/api.js), and only touch [src/pages/LoginPage.jsx](../src/pages/LoginPage.jsx) if the login page is also part of the sample preview or progress indicator flow. The implementation should keep raw keystrokes in the browser, convert them into summarized metrics before network submission, sync only after enough samples are collected, and ensure the profile-build page clearly shows when a sample has actually been persisted versus just collected locally.

## Prompt 3: System design hardening and user-facing flow

Use this prompt if you want Claude to redesign the end-to-end profile-build experience so it is safe for a multi-tenant SaaS. The goal is to make the architecture robust enough that onboarding, live trust scoring, and later drift checks do not interfere with each other. Please update [src/pages/ProfileBuildPage.jsx](../src/pages/ProfileBuildPage.jsx), [src/context/AuthContext.jsx](../src/context/AuthContext.jsx), [backend/main.py](../backend/main.py), [backend/database.py](../backend/database.py), [backend/models.py](../backend/models.py), and the documentation file [README.md](../README.md) if the user flow or stored data contract changes. The implementation should treat profile-build as a dedicated onboarding state, require an authenticated session and tenant/site context, store only aggregated behavioral patterns in MongoDB, and make the UI reflect a clear state machine such as collecting, syncing, stable, and drifted so that each user ends with a visible and queryable biometric pattern in `biometric_profiles`.

## What Success Looks Like

At the end of the flow, each user should have one MongoDB profile document that represents their stable behavioral pattern, plus optional dashboard summaries in `users`. The profile-build UI should progress from sample collection to stable baseline, and the backend should be the only place that decides when a profile is safe to persist.
